## Part 2: What We Learn From Atlas & AgentMemory

### From Atlas (Pure bash, file-based, simple)

| Concept | What Atlas Does | How Mnemo Should Adapt |
|---------|----------------|----------------------|
| **Fail-safe hooks** | Every line has `|| true`, always exits 0 | Our hooks MUST never break the agent session |
| **Warm memory synthesis** | At stop, creates structured session summary (prompts + tools + phases) | We should synthesize session summaries, not just extract "learnings" |
| **Repeated question detection** | Hashes normalized prompts, counts repeats, nudges persistence | Simple, effective — we could add this with 20 lines |
| **Learning detection** | Counts indicator words (≥2 = learning). Simple regex, no ML | Our intent classifier is overengineered — regex baseline first, ML on top |
| **File-per-learning** | Each learning is a standalone markdown file with YAML frontmatter | Easier to debug than a monolithic memory.json |
| **Search = grep** | Keyword extraction → grep across learning files | Too simple for us, but the principle of "just works" matters |
| **Global scope** | `~/.atlas/` not per-repo | We're per-repo which is better for isolation, but should support global prefs |

**Key Takeaway**: Atlas works because it's so simple it CAN'T break. Our system is complex enough that it breaks silently.

### From AgentMemory (TypeScript, daemon, LLM-dependent)

| Concept | What AgentMemory Does | How Mnemo Should Adapt |
|---------|----------------------|----------------------|
| **Daemon architecture** | Long-lived HTTP server on localhost:3111 | Our MCP server cold-starts per call — fine for now, but daemon could help |
| **Fire-and-forget hooks** | Hooks `fetch()` to daemon + `setTimeout(exit, 500)` — never blocks | Our hooks run inline. We should ensure they NEVER block the agent |
| **Observe everything** | Every hook POSTs raw observation. Processing happens server-side | We should capture raw events, process later (not inline in hooks) |
| **Working memory vs Archival** | Explicit pinned "core memory" (always shown) + decayed archival | Better UX than our single retention score. "Pinned" = always in context |
| **Crystallization** | At session end, LLM distills action chains → permanent lessons | We CAN'T use LLM (zero deps), but Kiro-as-classifier is the equivalent |
| **Project scoping** | Memories tagged with project ID, not tied to repo path | We're repo-scoped (good) but could tag memories with project context |
| **Multi-agent isolation** | Different agents get different memory views | Not needed now, but good architecture |
| **Session tracking** | Explicit session start/end with observation counts | We have no session concept — hooks fire independently |
| **12 hooks** | Including subagent-start/stop, task-completed, post-commit | We have 5. Missing: post-commit is useful |
| **Consolidation pipeline** | Periodic: merge similar memories → higher-order patterns | We have contradiction detection but not proactive merging |

**Key Takeaway**: AgentMemory's "observe everything, process later" model is correct. Our hooks try to be too smart at capture time — they should be dumb collectors.

## Part 3: The Correct Architecture

### Core Insight

The memory capture problem has ONE root cause: **hooks can't see enough context to make intelligent decisions in real-time**.

- `user-prompt-submit` sees only the user's message (not the agent's response)
- `post-tool-use` sees only the tool name and output (not what the agent learned from it)
- `stop` sees only the final assistant message (not the full session)

**Atlas solution**: Accept this limitation, do regex matching on what you CAN see, rely on the learning capture being "good enough" over many sessions.

**AgentMemory solution**: Run a daemon, capture everything as raw observations, use an LLM at session end to synthesize.

**Mnemo's correct solution** (for Kiro): 

```
During session:
  - Hooks capture raw events (fast, dumb, never fail)
  - user-prompt-submit: captures user message + searches relevant memories → inject
  - post-tool-use: captures tool+output (just stores, no intelligence)
  - No processing during session

At session end (stop hook):
  - Read .kiro/sessions/cli/<id>.jsonl (full session transcript)
  - OR read accumulated raw observations from hooks
  - Spawn kiro-cli --no-interactive to classify (Kiro-as-LLM)
  - Extract structured memories: CATEGORY|content
  - Write to memory.json

Next session (agent-spawn):
  - Read memory.json → inject hot memories into context
  - Read decisions.json → inject permanent decisions
  - Read active plan → inject task status
```

This gives us:
- **Atlas's simplicity**: hooks are dumb collectors, never break
- **AgentMemory's completeness**: we see the full session transcript
- **Our unique advantage**: deterministic graph + semantic search without LLM dependency

## Part 5: How We Execute

### Principle: One feature at a time, fully verified

For EACH task:
1. Read the existing code for that feature
2. Write a failing test that proves the feature works end-to-end
3. Fix the code until the test passes
4. Run full test suite — nothing regresses
5. Manually verify with a real `mnemo init` + tool call if applicable
6. Move to next task

### Order of execution:

```
Week 1: Tier 0 (Foundation) — all 4 tasks
Week 2: Tier 1 (Memory Capture) — tasks 1.1-1.4
Week 3: Tier 2 (Memory Quality) — tasks 2.1-2.5
Week 4: Tier 3 + Tier 4 (Polish + verify untested)
Week 5+: Tier 5 (New capabilities)
```

### What NOT to do:

- ❌ Don't add new features until existing ones work
- ❌ Don't write code without a test that proves it works
- ❌ Don't trust that "tests pass" means "feature works" (the hook tests passed but hooks were broken)
- ❌ Don't optimize for speed — optimize for correctness
- ❌ Don't refactor for aesthetics — refactor only when it's blocking a fix

## Part 6: Success Criteria

Mnemo is "working 100%" when:

1. **Memory persists across sessions** — demonstrated by: session 1 stores memory, session 2 recalls it without being told
2. **Hooks never break the agent** — every hook exits 0, never hangs, never produces garbage output
3. **Search finds relevant context** — semantic search returns related memories when asked about a topic
4. **Recall is compact and useful** — ~500 tokens of targeted, relevant context at session start
5. **Every shipped feature has a test** — no untested modules in the critical path
6. **`mnemo init` + agent session just works** — zero manual configuration beyond `mnemo init --client kiro`
